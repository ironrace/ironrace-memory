## Disagreement clusters

The 39 disagreements cluster on three systemic labeler issues:

### Cluster A — `pub` → `pub(crate)` mis-classified as `needs_revalidation` (15 rows)

Ripgrep narrowed visibility on many internal items (`pub struct Foo` → `pub(crate) struct Foo`) between T₀ and HEAD. For `PublicSymbol` facts, narrowed visibility means the public-symbol claim is no longer true — the SPEC-correct label is `stale_source_changed`. The labeler's `structurally_classifiable()` returns false on visibility-only transitions, so it punts to `needs_revalidation`.

Affected rows (sample): 122, 123, 124, 126, 128, 130, 132, 137, 139, 144, 151, 154, 155, 156, 157.

### Cluster B — RA workspace/symbol resolution against HEAD instead of per-commit tree (15+ rows)

The labeler spawns rust-analyzer **once** against the working tree at run time. When `Replay::run` walks per-commit and queries RA's `workspace/symbol`, RA answers using its in-memory model of the working tree (= HEAD), not the per-commit tree. Result: for symbols deleted after T₀ but before HEAD, `symbol_resolves()` returns true (RA finds them at HEAD), and the labeler punts to `needs_revalidation`. For symbols deleted but with a similarly-named sibling at HEAD, the rename heuristic incorrectly fires.

This is the known `TODO(phase-0b-pilot)` from PR #30 (`benchmarks/provbench/labeler/src/replay.rs:79`) called out in PR #31's "Out of scope / follow-ups."

Affected rows (sample): 82, 83, 109, 115, 116, 118, 119 (and overlaps with cluster C).

### Cluster C — `similar`-crate rename heuristic over-fires (10+ rows)

When the labeler can't resolve a symbol's qualified path, it asks the `similar` crate for a rename candidate by name-similarity within the same file/module. The current threshold accepts coincidental name overlaps (e.g., `all_verbatim_literal` → `any_literal`, `json_stats` → `json_*`-anything) as renames when the actual change was a deletion alongside surviving sibling fields. Result: real deletions are labeled `stale_symbol_renamed`.

Affected rows (sample): 3 (`AstAnalysis::all_verbatim_literal`), 9 (`Config::json_stats`), 11, 15, 16, 26, 52, 57, 79, 80, 103, 104, 106, 186, 187, 188.

The labeler correctly identified rename in **2** cases (rows 60, 61: `locations_mut` → `captures_mut` per `a6dbff5 regex: s/locations/captures`). The reviewer confirmed the rename via the same `git log -S<old_name>` trace that surfaces the rename commit. So the heuristic *can* be made discriminating.

### Cluster D — DocClaim context-shift false positive (1 row)

Row 1: `DocClaim::column::FAQ.md::217`. The claim sentence text is byte-identical at the labeled commit — only its line number shifted from 217 to 243 because earlier markdown grew. The labeler's span hash for `DocClaim` evidently includes surrounding context that *did* change (other paragraphs were edited), so the hash mismatches even though the specific claim is unchanged.

The SPEC-correct label is `valid` (the claim still holds), not `stale_source_changed`.

## Suggested labeler fixes (input to Phase 0b hardening pass 3)

In rough priority order:

1. **Cluster A — visibility-narrowing classification.** Extend `structurally_classifiable()` in `src/label.rs` to recognize visibility-token transitions (`pub` ↔ `pub(crate)` ↔ `pub(super)` ↔ `pub(in path)` ↔ private). Map to `stale_source_changed`. Lowest-risk fix; ~15 rows flip from disagree to agree.

2. **Cluster B — per-commit-tree symbol resolution.** Three viable approaches:
   - **B1.** Spawn RA per-commit in a temp worktree. Wall-clock cost is prohibitive at ~600 commits × ~30s RA warmup; rejected on principle.
   - **B2.** Replace RA's role in `Replay::run` with tree-sitter on the per-commit blob: query for the symbol's qualified path directly in the AST. Faster, deterministic, self-consistent with the T₀ extractor.
   - **B3.** Keep RA but reset its workspace and send `didChangeWatchedFiles` between commits; reuse the spawned process. Reduces startup cost but RA's incremental indexing is unreliable for our use case.

   **Recommended: B2.** Drop RA from `Replay::run` entirely; use tree-sitter qualified-path resolution. Affects ~15-19 rows.

3. **Cluster C — rename heuristic.** Raise the `similar` threshold OR add a structural-context check (same struct, same kind of item) before accepting a rename. The 2 true renames (rows 60/61) shared the same enclosing impl block — that's a usable signal. Affects ~10 rows.

4. **Cluster D — DocClaim span boundary.** Hash only the claim sentence text, not the surrounding paragraph context, so positional shifts don't break the hash. Affects 1 row in this sample but is structurally important for documentation-heavy repos.

After applying these fixes, re-run the corpus and re-spot-check on a fresh stratified sample of 200 rows. Target: point estimate ≥95%, Wilson 95% lower bound ≥90% (per SPEC §9.1).
