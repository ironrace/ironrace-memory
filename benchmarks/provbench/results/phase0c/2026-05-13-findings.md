# ProvBench Phase 0c — LLM-as-Invalidator Baseline Findings (2026-05-13)

**Status:** **SPEC §9.3 week-4 kill criterion SATISFIED on the pilot
corpus subset (44% coverage).** Cohen's κ between Sonnet 4.6's
invalidation decisions and the mechanical labeler is **−0.001 (CI
[−0.003, +0.002])** — statistically indistinguishable from chance.
The rule labeler dominates the LLM baseline on stale detection by
~227× recall at $0 incremental cost; the LLM-as-invalidator paid
**$2.59 per correct invalidation**.

This is the Phase 0c numeric output the SPEC §9.2 contract was chasing.
Coverage is honest "subset" (4,050 / 9,232 rows), not full — the run
budget-aborted at 95% of the $30 operational cap, exactly as designed.

## Parse-failure characterization (5/270 batches = 1.8%)

All five failures landed in
`results/phase0c/2026-05-13-canary/parse_failures.jsonl`. None were
API faults. Pattern breakdown:

| Pattern | Count | What the model emitted instead of `Vec<Decision>` JSON |
|---|---:|---|
| Pure prose explanation | 1 | Narrated the diff, listed which facts referenced which files, concluded with a sentence — no JSON. |
| Markdown code-fence wrapper | 2 | ` ```json\n[...]\n``` ` — valid JSON inside ``` fences, but `serde_json::from_str` fails on the fences. |
| Prose preamble + code-fenced JSON | 2 | Same fence problem, preceded by a paragraph of reasoning. |

These are prompt-compliance failures, not capability failures. They
would likely be recoverable by either:

1. A response post-processor that strips ``` fences and leading prose
   before `serde_json::from_str`, or
2. A stricter prompt that explicitly forbids prose preambles and
   markdown fencing on the parse-retry attempt.

Neither change was applied in this run; the 5 batches are honest
"refusals" under the frozen prompt.

## Hygiene flags (must address before any external write-up)

1. **Coverage is "subset" (44%).** `metrics.json` records this
   correctly. Do not present these numbers as satisfying SPEC §9.2's
   `coverage: "full"` contract — that gate is not yet met.
2. **Mixed-binary lineage.** Batches 1–21 were produced by the
   pre-patch baseline binary; batches 22–270 by patched
   `0b4d441` (`fix(provbench-baseline): skip-and-log on parse
   failure`). The patch only changes error-handling, not prompt or
   scoring — the prompt is byte-stable (`tests/prompt_frozen.rs`
   covers this). A SPEC §11 entry must record this.
3. **`baseline_crate_head_sha` in `manifest.json`** is the pre-patch
   SHA. The runner does not reverify it at runtime, so the on-disk
   manifest is stable but does not name the binary that produced
   batches 22–270.
4. **No held-out evaluation** has been run. SPEC §9.4 scale-out gate
   still pending serde + flask.

## What this result is (and isn't)

**Is:** evidence that, under the SPEC §6.1 frozen prompt and Sonnet 4.6
at temperature 0, an LLM-as-invalidator cannot replace a mechanical
labeler that operates on structural diff features. The κ CI is tight
enough that extending coverage will move the third decimal place, not
the verdict.

**Is not:** a claim about LLMs in general, or about Sonnet 4.6 on
related tasks. The SPEC §6.1 prompt is the artifact being evaluated;
a different prompt (more elaborate few-shot examples, chain-of-
thought, different output schema) could plausibly perform very
differently. Future work that wants to revisit this verdict should
re-freeze the prompt and re-run.
