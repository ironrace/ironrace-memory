# ProvBench Phase 1 (rules-based) — 2026-05-14 canary findings

**Status:** **SPEC §8 #3 / #4 / #5 cleared on the Phase 0c canary
subset.** A deterministic, structural, single-repo HEAD-only rule
chain (R1–R6 firing under `rule_set_version v1.0`) classifies 4,387 evaluation
rows in ~21 wall-seconds with a per-row p50 of 2 ms, recovering
2,401 of the 2,496 ground-truth stales the LLM-as-invalidator baseline
missed.

This is the Phase 1 numeric output the SPEC §8 contract was chasing
after Phase 0c satisfied the §9.3 week-4 kill criterion at
κ = −0.001, stale recall 0.004, $2.59 per correct invalidation.

## Thesis

A deterministic, structural, single-repo HEAD-only rules pass clears
SPEC §8 #3 (valid retention WLB ≥ 0.95), #4 (latency p50 ≤ 727 ms),
and #5 (stale recall WLB ≥ 0.30) verbatim over the Phase 0c canary
subset, replacing the LLM-as-invalidator (Phase 0c result:
κ = −0.001, stale recall 0.004). Stale recall additionally clears
the internal ≥0.80 stretch target (WLB 0.9537) as a bonus.

## Latency methodology

The `wall_ms` field name is identical between the LLM baseline
(`results/phase0c/2026-05-13-canary/predictions.jsonl`) and the
phase1 rules runner, but the granularity is NOT directly comparable:

- **Baseline (LLM):** per-batch `wall_ms` — one wall-clock value per
  Anthropic API round-trip covering many facts. The scoring crate
  dedupes by `batch_id`, sums per `commit_sha`, then takes a
  nearest-rank p50 over commits. The baseline column's
  `latency_p50_ms = 7267` is therefore a **per-commit median**
  (ms-to-seconds scale).
- **Phase 1 (rules):** per-row `wall_ms` — one wall-clock value per
  fact classification. The scoring crate takes a floor-median over
  per-row values. The candidate column's `latency_p50_ms = 2` is a
  **per-row median** (ms scale).

Phase 1 classifies a fact in median ~2 ms; the LLM baseline took
median ~7.3 s per commit. The §8 #4 threshold (≤ 727 ms) is on the
candidate column alone, which the rules runner satisfies with
margin. The
`metrics.json.deltas.latency_p50_ratio_baseline_per_commit_to_candidate_per_row`
figure of **3633.5** is a useful headline but is NOT a direct
apples-to-apples throughput comparison — readers should treat it as
an order-of-magnitude indicator under mixed units, not a literal
speedup ratio.

## Hygiene flags (must address before any external write-up)

1. **R4 line-presence probe is heuristic, not structural.** R4 looks
   for the labeled anchor line in the post-image and infers
   Valid/Stale from presence alone. The canary confusion matrix
   shows **17 GT=Valid → Stale (false positives)** and
   **95 GT=StaleSourceChanged → Valid (false negatives)** at R4.
   The probe passes the §8 gates on this canary, but does not
   provide a structural guarantee and could regress on held-out
   repos (§9.4 follow-up).
2. **R7 is unreachable via `RuleChain::default()`.** R1 fires first
   when `post_blob.is_none()`, masking R7 entirely. R7 cells in
   `per_rule_confusion` are absent for that reason. Documented in
   code; the gate clears without R7 firing. Future tuning may
   restructure the chain to re-enable R7.
3. **NR routing accuracy is effectively ~0 for both runners.**
   Phase 1's R1–R7 chain emits zero `NeedsRevalidation` verdicts on
   this canary (`RunStats { ..., needs_reval: 0 }`); the LLM baseline
   emits NR for 0.21% of rows. §8 does not gate NR routing; phase 1
   is no worse than the baseline on this axis. Re-routing through
   an LLM second-pass on R4-marginal rows is a v2 follow-up
   (out of scope here).
4. **Coverage is "subset" (pilot canary), not full-corpus.** The
   4,387-row subset reflects the Phase 0c baseline-run's row set.
   Held-out evaluation on `serde` and `flask` is deferred to
   SPEC §9.4. Do NOT present these numbers as satisfying §9.2's
   `coverage: "full"` contract — that gate is not yet met.
5. **Anti-leakage / SPEC §10:** pilot repo only. R3 and R4 rule
   semantics were tuned during Task 4 in response to this canary's
   results, which is the SPEC §10-permitted pilot tuning regime.
   SPEC §10 forbids held-out tuning; this result therefore must not
   be used as a held-out claim, and the held-out repos must be run
   under the frozen `rule_set_version v1.0` artifact recorded in
   `run_meta.json` and `phase1_git_sha = 0ffb67bd1f542acacd7d6d6756bdb2e44d851db4`.
6. **Determinism preserved.** Two re-runs of `provbench-phase1
   score` on identical inputs produce byte-identical
   `predictions.jsonl` and `rule_traces.jsonl` modulo the `wall_ms`
   field. Covered by `phase1/tests/determinism.rs`.

## What this result is (and isn't)

**Is:** evidence that, on the Phase 0c canary subset, a deterministic
rule chain operating purely on pre/post tree blobs and structural
diff features clears the SPEC §8 deployment thresholds at zero LLM
cost, with margins large enough that the third decimal place is not
in question. The result is consistent with SPEC §9.3's prediction
that a rule-based labeler dominates the LLM-as-invalidator on the
ProvBench task structure.

**Is not:** a claim about held-out generalization, cross-repo
robustness, or semantic-equivalence reasoning. The R4 probe in
particular is a heuristic and may regress on repos with different
diff structures. The held-out evaluation under SPEC §9.4 is the
load-bearing follow-up.

## Out of scope

- Cross-repo, tunnels, multi-branch (SPEC §12).
- Semantic-equivalence judgments (SPEC §3.2).
- Test-assertion facts (SPEC §3.1 #5).
- LLM-in-the-loop refinement of R4-marginal or NeedsRevalidation rows
  (v2 LLM second-pass).
- Integration into the ironmem runtime hot path (this run is the
  isolated ProvBench scaffold per `project_provbench_baseline_isolated`).
- SPEC §9.4 held-out repos (`serde`, `flask`).
