## What changed between v1.0 and v1.1

`v1.1` is a **structural rule-chain change with no §7.1 impact on this canary.** The per-rule confusion shifts 47 rows from R1 to R7:

| Rule | v1.0 fires | v1.1 fires | Δ |
|---|---|---|---|
| R1 `source_file_missing` | 783 | 736 | **−47** |
| R2 `blob_identical` | 240 | 240 | 0 |
| R3 `symbol_missing` | 824 | 824 | 0 |
| R4 `span_hash_changed` (line-presence probe) | 2,468 | 2,468 | 0 |
| R5 `whitespace_or_comment_only` | 62 | 62 | 0 |
| R6 `doc_claim` | 10 | 10 | 0 |
| R7 `rename_candidate` | 0 | 47 | **+47** |

All 47 R7 hits in v1.1 have `ground_truth == stale_source_deleted` and `prediction == stale`. In v1.0 these same 47 rows were caught by R1 (also `prediction == stale`). The Decision is the same in both versions — the change is in the **reason code** (R1's `stale_source_deleted` vs R7's `stale_symbol_renamed`) and the audit trail.

§7.1 rolls all stale flavors into one bucket for recall/precision/F1, so the macro numbers don't move when 47 rows shift between two stale-producing rules. The change is visible in `metrics.json.per_rule_confusion` and `rule_traces.jsonl`.

## Latency methodology (unchanged from v1.0)

The candidate column reports per-row `wall_ms` (rule-classification cost per fact, µs–ms scale). The LLM baseline column reports per-commit `wall_ms` (per-batch API round-trip, hundreds of ms to seconds). These are **not** directly comparable; the `latency_p50_ratio_baseline_per_commit_to_candidate_per_row` delta in `metrics.json` is a useful headline but is not an apples-to-apples speedup. The §8 #4 ≤ 727 ms threshold applies to the candidate column alone, which is satisfied with three orders of magnitude of margin.

## Hygiene flags

1. **R4 line-presence probe is still heuristic.** Same caveat as v1.0: 17 false-Stale on GT=Valid and 95 false-Valid on GT=stale_source_changed on this canary. Pilot-only tuning admitted by SPEC §10; flagged for §9.4 follow-up.
2. **R7 fires on a narrow class.** Single-token Jaccard between a symbol leaf and a file stem is binary (1.0 on exact match, 0.0 otherwise), so R7 only catches the "function moved to a same-name file with the same extension" case. A more sensitive rename detector that reads candidate-blob content is a follow-up if the canary's 47 R7 wins look representative on held-out repos.
3. **Needs_revalidation routing accuracy = 0.** Same as v1.0 and the LLM baseline (~0.2%); §8 does not gate NR routing.
4. **Coverage:** subset (the 4,387-row pilot canary), not full-corpus. Pilot repo only; §9.4 held-out evaluation is a follow-up.
5. **Anti-leakage:** v1.0 → v1.1 is a chain-order + heuristic change, not a threshold retune. SPEC §10 forbids held-out tuning; the v1.1 rule set is what the §9.4 evaluation must use, frozen at phase1 git SHA `ccfc901be171`.
6. **Determinism preserved:** the `phase1/tests/determinism.rs` contract still holds (byte-identical-modulo-`wall_ms` across re-runs). The R7 ordering + leaf-stem proxy did not introduce non-determinism.
7. **v1.0 record retained.** `results/phase1/2026-05-14-canary/` is **not** rewritten; the v1.0 artifact is the historical record. v1.1 lives at `2026-05-15-canary/` so the audit trail shows the rule-set evolution.
